เว็บโหลดช้าเสียเงินเปล่า: วิธีตรวจสอบและแก้ปัญหา Page Speed

Amazon พบว่าทุก 100ms ที่โหลดช้าลง ยอดขายหายไป 1% Google พบว่า 53% ของผู้ใช้มือถือออกจากเว็บถ้าโหลดนานกว่า 3 วินาที ความเร็วของเว็บไม่ใช่แค่ technical detail แต่เป็นเรื่องธุรกิจโดยตรง

ผลกระทบของเว็บโหลดช้าต่อธุรกิจ

ความช้าของเว็บไซต์ส่งผลกระทบต่อธุรกิจใน 3 มิติหลัก:

  • Bounce Rate สูงขึ้น — ผู้เข้าชมออกก่อนที่เว็บจะโหลดเสร็จ เสียโอกาสทางธุรกิจทันที
  • Conversion Rate ต่ำลง — แม้รอจนเว็บโหลด ผู้ใช้ที่หงุดหน่ายก็มีโอกาส convert น้อยกว่า
  • SEO Ranking แย่ลง — Google ใช้ Core Web Vitals ซึ่งรวม LCP เป็นปัจจัยจัดอันดับ

เครื่องมือตรวจสอบความเร็วเว็บไซต์

  • Google PageSpeed Insights (pagespeed.web.dev) — วิเคราะห์ทั้ง mobile และ desktop พร้อมคำแนะนำแก้ไขเฉพาะจุด
  • GTmetrix — วิเคราะห์ละเอียด พร้อม waterfall chart ของ resource ที่โหลด
  • WebPageTest — ทดสอบจาก location จริง รองรับ multiple test
  • Chrome DevTools → Network tab — ดู resource ที่โหลดแต่ละชิ้น

สาเหตุหลักที่ทำให้เว็บโหลดช้า

1. รูปภาพที่ไม่ได้ Optimize

รูปภาพมักเป็นสาเหตุหลักที่สุด หลักการ:

  • ใช้ format WebP แทน JPG/PNG — ขนาดเล็กกว่า 25–35%
  • Compress รูปก่อน upload — ใช้ Squoosh หรือ TinyPNG
  • ระบุ width และ height attribute เสมอ — ป้องกัน layout shift
  • ใช้ loading="lazy" สำหรับรูปที่อยู่นอก viewport
  • fetchpriority="high" สำหรับ hero image เพื่อ preload

2. JavaScript ที่หนักเกินไป

JavaScript ที่มีขนาดใหญ่หรือ execute นานจะ block main thread ทำให้ INP แย่และ page ดู "แช่แข็ง" ชั่วคราว วิธีแก้:

  • ใช้ defer หรือ async attribute สำหรับ script ที่ไม่จำเป็นต้องโหลดก่อน
  • Code splitting — โหลดเฉพาะ code ที่จำเป็นสำหรับหน้านั้นๆ
  • ลบ JavaScript ที่ไม่ได้ใช้งาน (dead code)
  • ลด dependency บน third-party script ที่ช้า

3. Server Response Time (TTFB) สูง

TTFB (Time to First Byte) วัดเวลาที่ browser รอได้รับ byte แรกจาก server เกณฑ์ที่ดีคือ <800ms สาเหตุหลัก:

  • Server อยู่ไกล — ใช้ CDN เพื่อ serve content จาก server ที่ใกล้ผู้ใช้
  • Shared hosting ที่แออัด — อัปเกรดหรือย้ายไป static hosting ที่เร็วกว่า
  • Database query ที่ช้า — ปัญหาหลักของ CMS เช่น WordPress

4. Render-Blocking Resources

CSS และ JavaScript บางชิ้นถูกโหลดก่อน render ทำให้หน้าเว็บแสดงผลช้า วิธีแก้:

  • Inline critical CSS — CSS ที่จำเป็นสำหรับ above-the-fold ให้ใส่ใน <style> tag โดยตรง
  • โหลด non-critical CSS แบบ async
  • ลดขนาด CSS/JS ด้วย minification

Static Site กับ Page Speed

Static website มีข้อได้เปรียบด้าน performance ตามโครงสร้าง เพราะไม่มี server-side processing ทำให้ TTFB ต่ำมาก และ serve ผ่าน CDN ได้ง่าย เว็บที่ WebExpressTH สร้างผ่าน PageSpeed Insights 90+ คะแนนเป็นมาตรฐาน

สรุป: Page speed เป็นทั้งปัญหา UX และปัญหา SEO พร้อมกัน จุดเริ่มต้นที่ง่ายที่สุดคือ optimize รูปภาพ — ได้ผลเร็วและเห็นการเปลี่ยนแปลงได้ชัด จากนั้นค่อยจัดการ JavaScript และ server configuration

เขียนโดย WebExpressTH — ทุกโปรเจกต์ของเรา optimize performance ตั้งแต่วันแรก

คำถามที่พบบ่อย

คะแนน PageSpeed เท่าไหรถึงจะดี?

90 ขึ้นไปถือว่าดีครับ 50–89 พอใช้ได้ ต่ำกว่า 50 ต้องปรับปรุงด่วน แต่คะแนน mobile มักต่ำกว่า desktop เพราะ mobile device มีพลังประมวลผลน้อยกว่า อย่าตกใจถ้าคะแนน mobile ต่ำกว่า desktop เสมอ

WordPress ทำให้คะแนน PageSpeed ต่ำจริงไหม?

บางส่วนใช่ครับ WordPress core มี overhead มากกว่า static site แต่ด้วย plugin cache ที่ดี, CDN และการ optimize ที่ถูกต้อง ก็สามารถทำคะแนนสูงได้ อย่างไรก็ตามต้องลงทุนเวลาและความรู้มากกว่า static site

CDN คืออะไรและจำเป็นไหม?

CDN (Content Delivery Network) คือเครือข่าย server ที่กระจายอยู่ทั่วโลก ทำให้ผู้เข้าชมได้รับ content จาก server ที่ใกล้ตัวที่สุด ลด latency ได้มาก สำหรับ static site บน Netlify หรือ Vercel CDN รวมอยู่ในแพ็กเกจแล้วครับ
